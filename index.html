Operating Systems, from first principles to the whiteboard.

Everything a computer does — running a browser, saving a photo, sending a keystroke — passes through an operating system. This guide walks you from the very basics through the topics that show up on interview whiteboards, with worked examples, diagrams, and a long Q&A section at the end.

01What is an Operating System?

An Operating System (OS) is a program that sits between your applications and the hardware. It is the boss that decides which program runs, when it runs, how much memory it gets, and how it talks to the disk, screen, network, and other devices.

Why do we need an OS at all?

  • Abstraction — an app doesn't care whether the disk is an SSD or a spinning platter; it just calls write().
  • Resource management — many programs want the CPU/RAM at once; the OS shares it fairly and safely.
  • Protection & isolation — a crashing app should not take down the whole computer or read another app's memory.
  • Convenience — file names, windows, sockets, threads — all niceties the OS provides.

Core functions

  1. Process management — create, schedule, and terminate processes.
  2. Memory management — allocate/free RAM, provide virtual memory.
  3. File management — organize files on storage, permissions.
  4. Device management — drivers for peripherals; I/O scheduling.
  5. Security & protection — users, groups, permissions, isolation.
  6. Networking — TCP/IP stack, sockets.
  7. User interface — CLI (shell) and GUI.
Analogy
Think of the OS as the manager of a busy restaurant. Diners are apps, tables are CPU cores, the kitchen is the disk, waiters are the drivers. Without a manager coordinating everyone, chaos.

03Kernel, User Mode, System Calls

User mode vs. kernel mode

A modern CPU has (at least) two privilege levels:

  • Kernel mode (ring 0) — can execute every instruction, touch every register, every byte of RAM.
  • User mode (ring 3) — restricted. Cannot access hardware directly, cannot see other processes' memory.

What is a system call?

A system call is a request from a user program to the kernel. On x86-64 Linux it is issued via the syscall instruction; the kernel jumps to a fixed handler that validates the request, performs it, and returns.

Common categories

CategoryPOSIX examples
Process controlfork, exec, exit, wait
Fileopen, read, write, close, lseek
Deviceioctl, read/write
Informationgetpid, time, uname
Communicationpipe, socket, send, recv, shmget
Protectionchmod, chown, umask

Kernel architectures

Monolithic Linux

Entire OS (scheduler, drivers, FS, networking) runs in one big kernel image at ring 0. Fast (no boundary crossings) but a driver bug can crash the kernel.

Microkernel MINIX, QNX

Only the bare minimum (IPC, scheduling, basic memory) runs in kernel space; drivers and file systems are user-space servers that talk via messages. Safer but slower.

Hybrid Windows NT, macOS

Microkernel-inspired structure but with performance-critical pieces (graphics, file cache) pulled into kernel space.

Exokernel research

Kernel does almost nothing; apps get raw hardware access via safe multiplexing. Extreme flexibility, extreme complexity.

04Processes

Program vs. process

A program is a passive file on disk (e.g. /bin/ls). A process is that program running — it has memory, registers, a stack, a program counter, open files, etc. One program can be running as many processes at once.

Address space of a process

Process states

Process Control Block (PCB)

The kernel keeps a PCB (a struct — task_struct in Linux) per process containing:

  • PID, parent PID, user/group IDs
  • Process state (running, ready, waiting, …)
  • CPU registers, program counter, stack pointer (saved on context switch)
  • Memory info: page tables, base/limit
  • Open file table, current working directory
  • Accounting info: CPU used, priority, nice value
  • Signal handlers, pending signals

Context switch

When the OS switches from process A to process B it must:

  1. Save A's registers + PC into A's PCB.
  2. Load B's registers + PC from B's PCB.
  3. Switch address space (change CR3 on x86 → flush TLB entries).

Cost is real: a few microseconds plus TLB/cache pollution afterwards. Too many switches ⇒ throughput dies.

Creating processes: fork() and exec()

pid_t pid = fork();          // returns twice
if (pid == 0) {              // child
    execvp("ls", args);      // replace image
    perror("exec"); _exit(1);
} else if (pid > 0) {        // parent
    int status;
    waitpid(pid, &status, 0);
} else {
    perror("fork");
}
  • fork() creates a near-identical copy of the current process (copy-on-write).
  • exec() replaces the current process's code+data with another program.
  • wait() lets a parent collect a child's exit status (otherwise the child becomes a zombie).
Interview classic
"How many processes exist after this code?" 
fork(); fork(); fork();
Answer: 2³ = 8 processes. Each fork doubles the count.

05Threads

A thread is an independent path of execution within a process. Threads in the same process share code, data, heap, and open files — but each has its own stack, registers, and program counter.

Process vs. thread — the interview table

ProcessThread
Address spaceOwnShared with siblings
Creation costHeavy (page tables, PCB)Light
Context switchExpensive (TLB flush)Cheap (same page table)
CommunicationIPC (pipe, socket, shm)Read/write shared memory directly
Fault isolationStrong — one crash won't kill anotherWeak — a crash takes the whole process

User-level vs. kernel-level threads

  • User-level (ULT) — managed entirely by a library in user space. Fast to create/switch, but if one thread blocks on a syscall, all siblings block. Cannot use multiple cores.
  • Kernel-level (KLT) — kernel knows about each thread and schedules them. Can use multiple cores; syscalls only block the caller. Slightly slower to switch.

Multithreading models

  • Many-to-One — N user threads → 1 kernel thread. Old Solaris green threads.
  • One-to-One — every user thread has its own kernel thread. Linux (NPTL), Windows.
  • Many-to-Many — pool of M kernel threads services N user threads.

06CPU Scheduling

When multiple processes are ready, which one runs next? That's the scheduler's job.

Key terms

  • Arrival time (AT) — when the process enters the ready queue.
  • Burst time (BT) — CPU time the process needs before its next I/O.
  • Completion time (CT) — when the process finishes.
  • Turnaround time (TAT) = CT − AT
  • Waiting time (WT) = TAT − BT
  • Response time = time from arrival until first CPU allotment.

Preemptive vs. non-preemptive

Non-preemptive: once a process gets the CPU it keeps it until it finishes or blocks. Preemptive: the OS can forcibly take the CPU away (via a timer interrupt).

Common algorithms

1. First-Come, First-Served (FCFS)

Simple FIFO queue. Fair-looking but suffers from the convoy effect — a long job at the head makes everyone else wait.

2. Shortest Job First (SJF)

Pick the process with the smallest burst next. Optimal for average waiting time — but requires knowing burst lengths in advance. Long jobs may starve.

3. Shortest Remaining Time First (SRTF)

Preemptive version of SJF. If a new arrival has a shorter remaining burst than the running process, preempt.

4. Priority Scheduling

Each process has a priority; highest priority runs. Risk: starvation of low-priority jobs — solved by aging (gradually raising the priority of waiting processes).

5. Round Robin (RR)

Each process gets a fixed time quantum. When it expires, the process goes to the back of the queue. Good for interactive systems. Quantum too small ⇒ context-switch overhead dominates; too big ⇒ becomes FCFS.

6. Multilevel Queue

Split ready queue into several queues (system, interactive, batch). Each queue may use a different algorithm and have a fixed priority.

7. Multilevel Feedback Queue (MLFQ)

Processes can move between queues based on observed behaviour: a job that uses its whole quantum is demoted (probably CPU-bound), one that blocks quickly is promoted (probably interactive). This is roughly what real UNIXes use.

Worked example — SJF (non-preemptive)

Average WT = (0+5+3+6)/4 = 3.5. Note at t=7, P3 (BT 1) is picked over P2 (BT 4) because SJF looks at remaining burst.

07Process Synchronization

The problem: race conditions

When two threads share data and at least one writes, the outcome depends on timing. Classic example:

// shared: int counter = 0;
// two threads run this 100,000 times each:
counter = counter + 1;
// expected final value: 200,000
// actual: something less, non-deterministically

Why? counter = counter + 1 is not atomic. It's really three steps: load, add, store. Interleavings lose updates.

Critical section

A critical section (CS) is code that touches shared state and must run atomically with respect to other threads' critical sections. A correct solution must satisfy:

  1. Mutual exclusion — at most one thread in the CS at any time.
  2. Progress — if no one is in the CS, a thread that wants in eventually gets in.
  3. Bounded waiting — no thread waits forever while others cycle through.

Peterson's algorithm (two-process, software-only)

bool flag[2] = {false, false};
int  turn;

// thread i:
flag[i] = true;
turn = 1 - i;
while (flag[1-i] && turn == 1-i) { /* spin */ }
// --- critical section ---
flag[i] = false;

Beautiful proof-of-concept but not used in real kernels because modern CPUs reorder memory writes.

Hardware primitives

  • Test-and-Set (TAS) — atomically read a word and set it to 1, returning the old value.
  • Compare-and-Swap (CAS) — atomically: if memory equals expected, replace it with new. Used to build lock-free data structures.

Semaphores Dijkstra, 1965

A semaphore is an integer with two atomic operations:

wait(S) / P(S):  while (S <= 0); S--;
signal(S) / V(S): S++;
  • Binary semaphore — value ∈ {0, 1}. Acts like a lock.
  • Counting semaphore — any non-negative integer; models N interchangeable resources.

Classic problems

Producer–Consumer (bounded buffer)

semaphore empty = N;   // free slots
semaphore full  = 0;   // filled slots
mutex     m;

// producer:
wait(empty); wait(m);
  buffer[in] = item; in = (in+1) % N;
signal(m); signal(full);

// consumer:
wait(full); wait(m);
  item = buffer[out]; out = (out+1) % N;
signal(m); signal(empty);
Trap
Swapping the order to wait(m); wait(empty); can deadlock — the consumer holds the mutex while waiting for an empty slot the producer needs the mutex to fill.

Readers–Writers

Many readers can be in the CS at once, but a writer needs exclusive access. Variants prioritise readers (writers may starve), writers (readers may starve), or fair.

Dining Philosophers

5 philosophers around a table, 5 forks between them; a philosopher needs both neighbouring forks to eat. Naïve solution deadlocks if everyone picks up their left fork at once.

Fixes: (a) let one philosopher pick up right first; (b) allow only 4 to try at once; (c) pick up both forks atomically.

08Deadlocks

A deadlock is a state where every process in a set is waiting for a resource held by another process in the same set — nobody moves.

Coffman conditions — all four must hold

  1. Mutual exclusion — at least one resource is non-shareable.
  2. Hold and wait — a process holding a resource can request another.
  3. No preemption — resources can only be released voluntarily.
  4. Circular wait — a cycle of processes each waiting for the next.

How to handle deadlocks

  1. Prevention — design the system so at least one Coffman condition can't hold. E.g. require all resources up front (no hold-and-wait), or number resources and require ordered acquisition (no circular wait).
  2. Avoidance — at runtime, only grant a request if the resulting state is still safe. Banker's Algorithm (Dijkstra) is the textbook example.
  3. Detection & recovery — let deadlocks happen, run a detector periodically, kill/rollback processes to break the cycle. Databases do this.
  4. Ignore it — the "ostrich algorithm." UNIX largely uses this: deadlocks are rare enough that reboots are cheaper than the machinery to prevent them.

Banker's Algorithm — sketch

System tracks:

  • Available[m] — free instances of each resource type.
  • Max[n][m] — max each process could ever request.
  • Allocation[n][m] — currently held.
  • Need = Max − Allocation.

On a request, tentatively grant it and run the safety algorithm: is there an ordering of processes such that each can finish given Available plus what earlier finishers release? If yes → grant. If no → block.

Deadlock vs. starvation vs. livelock

  • Deadlock — waiting forever, holding a resource.
  • Starvation — waiting forever, holding nothing (scheduler keeps ignoring you).
  • Livelock — processes are running but making no progress (e.g. two people politely stepping aside for each other in a corridor).

09Memory Management

Goals

Keep multiple processes in RAM safely, give each the illusion of a large private address space, and manage the physical DRAM efficiently.

Contiguous allocation

Each process gets one contiguous chunk of RAM. Simple, but leads to fragmentation:

  • External fragmentation — plenty of free RAM total, but no single hole is big enough.
  • Internal fragmentation — allocated block is bigger than what the process actually uses.

Allocation strategies

  • First fit — first hole big enough. Fast.
  • Best fit — smallest hole that fits. Wastes the least, but creates tiny slivers.
  • Worst fit — largest hole. Rarely useful.

Paging

Break physical memory into fixed frames and logical memory into equal-sized pages. Any page can go in any frame — no external fragmentation.

TLB — Translation Lookaside Buffer

Consulting the page table on every load/store would be crushingly slow. The MMU has a small associative cache of recent translations — the TLB. Hit rate is usually 95–99%. On a context switch, part or all of the TLB is invalidated (or tagged with an ASID).

Segmentation

Divide memory by logical unit (code, data, stack) rather than fixed size. Each segment has a base + limit. Modern systems combine paging with segmentation, or just use paging.

Multilevel & inverted page tables

  • A flat page table for a 48-bit address space is huge → use hierarchical tables (x86-64 uses 4 or 5 levels).
  • Inverted page table has one entry per frame, not per page — good for very large virtual spaces (used on PowerPC, IA-64).

10Virtual Memory

Virtual memory gives each process an address space larger than physical RAM by treating disk (swap) as an overflow area, and loading pages on demand.

Demand paging

Don't load a page until it's accessed. Each page-table entry has a valid bit; accessing an invalid page triggers a page fault. The OS then:

  1. Traps into the kernel.
  2. Finds a free frame (evicting one if needed).
  3. Reads the page from disk into the frame.
  4. Updates the page table, restarts the faulted instruction.

Page-replacement algorithms

FIFO

Evict the oldest-loaded page. Simple but suffers from Belady's anomaly: more frames can, paradoxically, cause more faults.

Optimal (OPT / MIN)

Evict the page that will be used furthest in the future. Theoretical benchmark — you can't know the future in general.

LRU — Least Recently Used

Evict the page not used for the longest time. Approximates OPT well; exact LRU is expensive in hardware (needs per-access timestamps), so real OSes use approximations.

Clock (Second-Chance)

Cheap LRU approximation. Frames arranged in a circle; each has a "reference" bit set on access. Sweeper hand walks around; if ref=1 clear it and skip, if ref=0 evict.

LFU / MFU

Least/Most Frequently Used — count accesses. LFU keeps historically hot pages even after they cool.

Belady's anomaly — worked example

Reference string: 1 2 3 4 1 2 5 1 2 3 4 5. With FIFO and 3 frames → 9 faults; with 4 frames → 10 faults. More RAM, more faults. Only FIFO exhibits this; stack algorithms (LRU, OPT) never do.

Thrashing

When a process (or the system) spends more time paging than executing. Symptoms: CPU utilisation drops even as the run queue grows. Cause: aggregate working set exceeds physical memory. Cures:

  • Reduce degree of multiprogramming (swap some processes out entirely).
  • Use a working-set model — keep only enough pages per process to hold its recent locality.
  • Use page-fault frequency control — if PFF too high, give the process more frames.

Memory-mapped files & copy-on-write

  • mmap maps a file directly into a process's address space — reads and writes become memory operations.
  • Copy-on-write — after fork(), parent and child share physical pages read-only; the first write triggers a copy. Saves huge amounts of memory in shell-heavy workflows.

11File Systems

What a file system provides

  • Named, hierarchical storage of persistent data.
  • Permissions and ownership.
  • Free-space tracking.
  • Crash-consistency (via journaling, copy-on-write, etc.).

File attributes (typical POSIX)

Name, inode number, type (regular, dir, symlink, device), size, permissions (rwx for user/group/other), timestamps (atime, mtime, ctime), owner UID/GID, link count.

Directory structures

  • Single-level — one flat namespace. Doesn't scale.
  • Two-level — per-user directory. Solves name collisions but no grouping.
  • Tree — the familiar hierarchy.
  • DAG — allow hard links (multiple names for same inode). Care needed for garbage collection.

File allocation methods

MethodIdeaProsCons
ContiguousFile occupies consecutive blocks.Fast sequential + random access.External fragmentation; hard to grow files.
LinkedEach block points to the next.No fragmentation; easy growth.Random access = O(n); pointer overhead; one bad block kills tail.
Indexed (inode)Index block lists all data-block numbers.Fast random access; no fragmentation.Index block itself uses space; multi-level for big files.

The UNIX inode

Free-space management

  • Bitmap — one bit per block. Compact, easy to find contiguous runs.
  • Linked list of free blocks — no wasted bits but slow scans.
  • Grouping / counting — record starts + lengths of free runs.

Journaling & copy-on-write

  • Journaling (ext3/ext4, NTFS) — write intended changes to a log first; if a crash interrupts, replay the log on reboot. Modes: journal (data+meta), ordered, writeback.
  • Copy-on-write (ZFS, btrfs, APFS) — never overwrite in place; write new versions and swing pointers atomically. Enables cheap snapshots.

The I/O subsystem

  • Buffering — smooths speed mismatch between producer and consumer.

13Advanced Topics

Virtualization

A hypervisor runs multiple guest operating systems on one physical machine.

  • Type-1 (bare-metal) — hypervisor runs directly on hardware. E.g. VMware ESXi, Xen, Hyper-V, KVM (Linux kernel module).
  • Type-2 (hosted) — runs as an app on a host OS. E.g. VirtualBox, VMware Workstation.

Hardware helpers: Intel VT-x, AMD-V add a "ring −1" so the hypervisor can safely trap sensitive guest instructions.

Containers

Not full VMs. A container is a group of processes sharing the host kernel but isolated via kernel features: namespaces (PID, mount, network, UTS, IPC, user) and cgroups (resource limits). Startup in milliseconds vs. seconds for a VM. Docker, containerd, LXC.

Real-Time OS

Guarantees that a task will meet its deadline. Techniques include:

  • Deterministic scheduling (Rate-Monotonic, Earliest-Deadline-First).
  • Preemptible kernel, bounded interrupt latency.
  • Priority inheritance to prevent priority inversion.

Priority inversion — the Mars Pathfinder story

Low-priority task L holds a mutex; medium task M preempts L; high-priority task H blocks waiting for the mutex L holds; L can't run because M is running; effectively H is delayed by M. Priority inheritance — temporarily boost L's priority to H's while it holds the mutex — fixes it.

Distributed OS ideas

  • Global naming, distributed file systems (NFS, Ceph).
  • Clock synchronization — Lamport clocks, vector clocks.
  • Consensus — Paxos, Raft.

Security basics

  • Authentication — who are you? (passwords, keys, biometrics).
  • Authorization — what are you allowed to do? (ACLs, capabilities).
  • Isolation — user vs. kernel mode; per-process address spaces; sandboxes.
  • Threats — buffer overflows, TOCTOU races, privilege escalation, side channels (Spectre/Meltdown).

14Interview Q&A

A curated set of questions that recur in interviews, grouped by topic. Read the question, try to answer aloud, then expand for the model answer.

Fundamentals

What is an operating system?

A system program that manages hardware resources (CPU, memory, I/O, storage) and provides services to application programs. It abstracts hardware, allocates resources fairly, isolates processes, and exposes convenient interfaces like files, processes, and sockets.

What is the difference between a program and a process?

A program is a static file on disk containing instructions. A process is a running instance of that program — with its own memory, registers, stack, PC, open files, and OS bookkeeping (PCB). One program can spawn many processes.

Kernel vs user mode — why do we need both?

Kernel mode can execute any instruction and access any memory. User mode restricts what the CPU allows so a buggy or malicious app can't corrupt the kernel or other processes. Every "dangerous" operation (I/O, memory-mapping, kernel data) must go through a system call, which is the only sanctioned way to switch modes.

What is a system call and how does it work?

A controlled entry point into the kernel. The user program puts a syscall number in a register, executes the syscall/int 0x80/svc instruction, which traps to a kernel handler. The handler validates arguments, performs the work, sets a return value, and returns to user mode.

Monolithic vs microkernel — trade-offs?

Monolithic: fast (in-kernel calls are cheap), but one buggy driver can crash everything. Microkernel: safer and more modular (drivers run as user processes talking via IPC), but message passing adds overhead. Real systems compromise — Linux is monolithic with loadable modules; Windows NT and macOS are hybrid.

Processes & threads

What information is stored in a PCB?

Process state, PID, program counter, CPU registers, memory management info (page tables), scheduling info (priority, queue pointers), accounting (CPU time used), I/O status (open files, pending I/O), and signal handlers.

What is a context switch and what does it cost?

Save the running process's CPU state to its PCB, load another process's CPU state, and switch page tables. Costs a few microseconds directly plus indirect costs — cold caches, flushed TLB. Modern kernels use ASIDs so the TLB isn't fully flushed on every switch.

Difference between process and thread?

Processes have separate address spaces; threads within a process share code, data, heap, and open files but have their own stack and registers. Threads are cheaper to create and switch, but a bug in one thread can crash the whole process.

What is a zombie process? An orphan?

Zombie: a child that has exited but whose parent hasn't called wait(). The kernel keeps the PCB around so the exit status is available. Orphan: a child whose parent died first — reparented to init (PID 1), which cleans up.

Describe fork() and exec().

fork() creates a copy of the calling process; returns the child's PID to the parent and 0 to the child. exec() replaces the calling process's image with a new program — same PID, new code/data/stack. The shell uses fork + exec to run every command.

User-level vs kernel-level threads?

User-level threads are scheduled by a library in user space — fast to create and switch, but a blocking syscall stalls all sibling threads and they can't use multiple cores. Kernel threads are known to the kernel — heavier but truly parallel and independently schedulable.

Scheduling

Which scheduling algorithm gives the minimum average waiting time?

Shortest Job First (SJF) — provably optimal for average waiting time. Downside: requires knowing burst times, and long jobs may starve.

Round-Robin — how do you choose the time quantum?

Too small → context-switch overhead dominates. Too big → RR degenerates into FCFS and interactive response suffers. Rule of thumb: quantum should be large enough that ~80% of CPU bursts finish within one quantum (typically 10–100 ms).

What is the convoy effect?

Under FCFS, a long CPU-bound job at the front of the ready queue makes many short jobs wait behind it — like cars stuck behind a truck. Results in poor average waiting time and CPU underutilisation while short jobs eventually block on I/O.

What is priority inversion and how do you fix it?

A high-priority task is blocked waiting for a lock held by a low-priority task, while a medium-priority task hogs the CPU — so effectively the medium task delays the high task. Fixes: priority inheritance (temporarily boost the lock holder), priority ceiling (locks are given a priority ≥ any potential holder).

Synchronization

What is a race condition?

A situation where the outcome depends on the non-deterministic interleaving of concurrent operations on shared state. Fixed by ensuring the critical section runs atomically (locks, atomics).

Semaphore vs mutex?

A mutex is a locking primitive with ownership — only the thread that locked it may unlock, and it often supports priority inheritance. A semaphore is a counter with wait/signal — no ownership; typically used for signalling (a producer signals when data is ready).

Spinlock vs blocking lock — when do you use which?

Spinlock: busy-wait; only correct when critical section is very short and on a multi-core system (otherwise you spin on the only CPU that could release the lock). Common inside kernels. Blocking lock: put the thread to sleep and let the scheduler wake it — use when the critical section might take longer than a context switch (~a few μs).

Explain the producer-consumer problem.

One or more producers put items in a bounded buffer; consumers take them out. Solutions coordinate three things: mutual exclusion on the buffer, "buffer not full" for producers, and "buffer not empty" for consumers — typically with two counting semaphores and one mutex.

What is the dining philosophers problem showing?

It illustrates deadlock and starvation in concurrent programs where processes need multiple shared resources. Naïvely, all philosophers can grab their left fork at once and starve. Classic solutions: acquire forks in a global order, limit concurrent attempts, or make fork pickup atomic.

Deadlocks

List the four Coffman conditions for deadlock.

Mutual exclusion, hold and wait, no preemption, circular wait — all four must hold simultaneously.

How would you prevent deadlock?

Break one of the four conditions. Common: eliminate circular wait by imposing a global ordering on resource acquisition. Or eliminate hold and wait by requiring processes to request all resources up front.

Explain Banker's algorithm briefly.

A deadlock-avoidance algorithm. Track Max/Allocation/Available/Need matrices. Before granting a request, tentatively allocate and check if the resulting state is safe — i.e. there exists a sequence in which every process can eventually finish. If yes, grant; if no, block the requester.

Deadlock vs starvation vs livelock?

Deadlock: processes stuck waiting for each other. Starvation: a process is repeatedly overlooked by the scheduler while others progress. Livelock: processes execute constantly but don't make forward progress (mutual retry loops).

Memory management

Internal vs external fragmentation?

Internal: an allocated region is larger than requested — the leftover inside the block is wasted (common with fixed-size allocations, paging). External: free memory exists but is broken into non-contiguous holes too small to satisfy a request (common in variable-size contiguous allocation).

What is paging?

Physical memory is divided into fixed-size frames; the logical address space into equal-size pages. Any page can map to any frame via a per-process page table. Eliminates external fragmentation; some internal fragmentation on the last page.

What is a TLB and why does it matter?

Translation Lookaside Buffer — a small associative CPU cache of recent virtual→physical page-table lookups. Without it, every memory access would require walking multi-level page tables. A high TLB hit rate is critical for performance; context switches and huge working sets hurt it.

Paging vs segmentation?

Paging uses fixed-size units; segmentation uses variable-size, logically meaningful regions (code, data, stack). Paging avoids external fragmentation; segmentation matches program structure but suffers from external fragmentation. Modern systems mostly use paging (sometimes on top of a small segmentation layer).

Virtual memory

Explain demand paging.

Only load a page into memory when it is first referenced. The page-table entry is marked invalid until then; access triggers a page fault, the OS reads the page from disk, updates the entry, and restarts the instruction.

Compare FIFO, LRU, and Optimal page replacement.

FIFO: evict oldest — cheap, can exhibit Belady's anomaly. LRU: evict least-recently-used — good approximation of temporal locality, expensive to implement precisely. Optimal: evict the page used furthest in the future — theoretical best; used only as a benchmark since it needs future knowledge.

What is thrashing? How do you detect and cure it?

The system spends most of its time paging rather than executing. Detect by monitoring page-fault rate and CPU utilisation (utilisation drops while faults spike). Cure by reducing the degree of multiprogramming, giving each process enough frames for its working set, or adding RAM.

What is Belady's anomaly?

The counter-intuitive phenomenon in FIFO page replacement where increasing the number of frames can increase the number of page faults for certain reference strings. Stack algorithms like LRU and OPT are immune.

What is copy-on-write?

After fork(), parent and child share the same physical pages, marked read-only. On a write, the MMU faults; the kernel duplicates that page and gives the writer a private copy. Saves memory and time when most pages are never written by the child (especially when the child immediately execs).

File systems & I/O

What is an inode?

A metadata record for a file in UNIX-like file systems. Contains permissions, owner, size, timestamps, link count, and pointers (direct + indirect) to the file's data blocks. Directory entries map file names to inode numbers.

Hard link vs symbolic link?

Hard link: another directory entry pointing to the same inode; can't cross file systems; deleting one name doesn't remove the file until link count = 0. Symlink: a special file whose contents are a path to another file; can cross file systems and point to directories; becomes dangling if target is deleted.

Contiguous vs linked vs indexed file allocation?

Contiguous: fastest access, worst fragmentation. Linked: no fragmentation, terrible random access. Indexed: good random access, some overhead for the index block. UNIX uses indexed (multi-level via indirect blocks).

What is a journaling file system?

A file system that first writes intended changes to an on-disk log ("journal") before applying them to the main data structures. If the system crashes mid-update, the journal is replayed on reboot to restore consistency.

Compare SSTF and SCAN disk scheduling.

SSTF: always serve the nearest request — minimises average seek but can starve distant requests. SCAN (elevator): head sweeps in one direction serving requests, reverses at the end — better fairness, no starvation, slightly worse average seek than SSTF for the general case.

Miscellaneous / advanced

What is DMA?

Direct Memory Access — a hardware feature letting a device transfer data to/from main memory without going through the CPU. The CPU sets up the transfer and gets an interrupt when it's done — freeing it to do other work.

Interrupt vs trap vs exception?

Interrupt: asynchronous event from hardware (timer, I/O completion). Trap: intentional synchronous switch to kernel (syscall). Exception: synchronous fault caused by an instruction (divide-by-zero, page fault). All three transfer control to a kernel handler.

Container vs virtual machine?

A VM emulates hardware and runs a full guest OS on a hypervisor; heavy but strong isolation. A container is a set of processes on the host kernel isolated by namespaces + cgroups; startup in milliseconds, but all containers share one kernel (weaker isolation).

What does a shell do?

Read a command line, parse it, resolve the program, fork(), exec() the program in the child, and wait() for it in the parent (unless backgrounded). Handles pipelines by wiring up file descriptors between children with pipe() before exec.

How does the OS handle a page fault, step by step?
  1. MMU detects an access to a page with valid bit clear → raises page-fault exception.
  2. CPU traps to the kernel; the fault handler runs.
  3. Handler determines if the address is legal (e.g. within the stack or a mapped region). If not, kill the process (SIGSEGV).
  4. Find a free physical frame — evicting one if necessary (page replacement).
  5. If the victim frame is dirty, write it to swap.
  6. Read the requested page from disk (or zero it for a new anon page).
  7. Update the page-table entry, set valid.
  8. Return from trap; restart the offending instruction, which now succeeds.
Explain how malloc gets memory from the OS.

malloc (in glibc) manages a heap in user space. For small allocations it hands out pieces of a large chunk grown via the brk()/sbrk() syscall (extends the process's data segment). For large allocations it calls mmap() to map an anonymous region. Freed memory is kept in free lists for reuse; only occasionally returned to the OS.

What are cgroups and namespaces?

Linux features that together make containers possible. Namespaces isolate what a process can see: PIDs, mounts, network interfaces, hostnames, users, IPC. cgroups limit and account resources: CPU, memory, block I/O, PIDs.

Describe how ls | grep foo works under the hood.

The shell calls pipe() to get a pair of fds. It forks twice. Child 1 dup2's the write end of the pipe to fd 1 and execs ls. Child 2 dup2's the read end of the pipe to fd 0 and execs grep foo. The shell closes both pipe fds in itself and waits. ls writes to its stdout — actually the pipe — and grep reads from its stdin — the other end.

Difference between preemptive and cooperative multitasking?

Preemptive: the OS forcibly interrupts running tasks (usually via a timer) to switch. Cooperative: tasks voluntarily yield the CPU; a task that never yields hangs the system. Modern OSes are preemptive; classic Mac OS <10 and Windows 3.x were cooperative.

What is a Real-Time OS and how is it different?

An OS that guarantees response within a bounded time. Hard RTOS (avionics, medical) — missing a deadline is a failure. Soft RTOS (video streaming) — missing a deadline degrades quality but is tolerable. Features: preemptible kernel, priority-based scheduling (RM, EDF), priority inheritance, bounded interrupt latency, no demand paging.

What is memory-mapped I/O?

Device registers appear at specific memory addresses; reading or writing those addresses talks to the device. Contrast with port-mapped I/O (special in/out instructions). Memory-mapped I/O is used by most modern architectures and lets code use ordinary load/store instructions.

Explain how the OS boots.
  1. Power on → firmware (BIOS/UEFI) runs POST and initialises hardware.
  2. Firmware loads the bootloader from a known location (MBR / EFI partition).
  3. Bootloader (GRUB, systemd-boot) loads the kernel image + initrd into memory and jumps to the kernel entry point.
  4. Kernel sets up paging, interrupts, drivers, mounts root filesystem, starts PID 1 (init/systemd).
  5. Init starts user-space services and eventually the login manager.
